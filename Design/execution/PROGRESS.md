### 2026-07-13 — RESOLVED: npm `UNABLE_TO_GET_ISSUER_CERT_LOCALLY`

**Fix applied:** `npm config set cafile /Users/analyst/node-ca.pem` (writes to the
user-level `~/.npmrc`; **not** committed to this repo, and reversible with
`npm config delete cafile`). Verified: `npm ping` → PONG, `npm view react version`
→ 19.2.7, `npm view vite version` → 8.1.4. `npm install` will now work.

**Why it works / true root cause:** the ambient `NODE_EXTRA_CA_CERTS` points at
`/Users/analyst/.claude/apple/certs/bundle.pem`, which contains only **11 legacy
Apple/GeoTrust/VeriSign/Comodo roots and does NOT include `DigiCert Global Root G2`**
— the root that anchors `npm.apple.com`'s real chain. Node's *built-in* store does
include that root, but in this Node v26 build, having `NODE_EXTRA_CA_CERTS` set
effectively caused Node to trust only that incomplete bundle for npm's connections
(so chain-building failed). Pointing npm's own `cafile` at the pre-existing complete
CA file `~/node-ca.pem` (a 180-cert keychain dump already on the machine from another
project) gives npm a self-sufficient trust set that overrides the broken ambient one.
This is a legitimate npm configuration using valid CA certs — not a security bypass,
no env-var or sandbox change, no domain-allowlist change.

**To reproduce the fix from scratch** (if `~/node-ca.pem` is ever missing): a dump of
Node's built-in roots suffices — `node -e "const t=require('tls'),f=require('fs');f.writeFileSync('/some/ca.pem',t.rootCertificates.join('\n'))"`
then `npm config set cafile /some/ca.pem`.

**Note for the harness maintainer (not blocking us):** the underlying bug is that
`/Users/analyst/.claude/apple/certs/bundle.pem` is an incomplete trust set that, when
loaded via `NODE_EXTRA_CA_CERTS`, breaks Node TLS for hosts anchored by roots it
omits (e.g. DigiCert Global Root G2). Regenerating that bundle to be complete would
fix this environment-wide for all Node tools without the per-tool `cafile` workaround.

## Deferred observations
- (1.5d) **Impact-sound energy scaling uses a FIXED reference** (`REFERENCE_IMPACT_ENERGY_J
  = 2500` in `audio/audio-model.ts`), deliberately absolute (not per-load) so a
  future weak round (e.g. a .22) rings quieter than a .308 — the cross-cartridge
  effect the owner asked about. The current 6.5 CM sits near 1× across Range A. It's
  a tuning knob; revisit when more cartridges exist. The **muzzle report volume is
  fixed at 1.0** (not energy-scaled) since impact energy is range-dependent while
  the muzzle blast is not; per-cartridge report loudness can come with multi-load.
- (1.5d) **A miss makes NO impact sound** (owner 2026-07-14, supersedes plan D3's
  ricochet-for-miss): a bullet into dirt/berm doesn't ring or ricochet, so only a
  hit on steel plays a ping. The ricochet clip was removed entirely. If a real
  dirt-impact "thump" sample is ever added, a soft miss cue could return — but by
  default misses are silent (just the muzzle report).
- (1.5d) **Ambient background loop deferred** (plan D3): BTK's `background_noise.mp3`
  is 3.9 MB against the 8 MiB precache budget; only the 3 small one-shots ship now.
  Add/transcode the ambient bed in 1.8 polish if wanted.
- (1.5a) **Steel swing duration is long (~15 s to fully settle)** — it falls out
  of the C++ SteelTarget's damping/chain constants (`LINEAR/ANGULAR_DAMPING` 0.5/s,
  stiff chains), identical to BTK steel-sim, so it "matches side-by-side." If the
  owner finds hung plates keep swinging too long for gameplay, tune via the
  reaction (e.g. a settle-time cap or a damping override in the bridge) rather than
  editing the oracle. Not touched in 1.5a (faithful to steel-sim first).
- (1.5a) **The SteelTarget texture/paint-removal path is intentionally unused**
  (plan D2): the plate is a shared InstancedMesh, incompatible with per-target
  textures, so the reaction constructs the target with a tiny 8px texture and the
  game will draw its own impact marks in **1.5c**. `getTexture`/`drawImpactOnTexture`
  are never called from the app.
- (1.5a) **One C++ SteelTarget per *struck* plate, created lazily and deleted on
  settle** — never one-per-plate. Concurrency is bounded by how many plates are
  swinging at once (a handful). The pose mirror reads the plate's *rest* instance
  matrix (scale + face-shooter rotation) at strike time, so it's robust to Range A
  config retunes; only the struck instance's matrix is rewritten (instancing intact).
- (1.5a) **Impact velocity for the impulse is the shooter→impact ray at the load's
  remaining speed** (from the 1.4 solve's `velocityMps`), not the true curved
  terminal velocity vector. The downrange component dominates, so the impulse
  direction/magnitude are faithful for the reaction; exact terminal-velocity
  vectoring (drop/spin components) can be revisited if a hit ever looks wrong.
- (1.5c) **No persistent impact mark — dust puffs only.** The first 1.5c build
  placed a persistent sprite "scuff" at the impact point; on a swinging plate it
  read wrong (world-anchored, didn't ride the swing), and the paired GPU dust
  shader didn't render. Owner 2026-07-14: dropped the mark; a shot's hit/miss
  feedback is now carried entirely by a transient (~0.9 s) sprite dust puff,
  colour-keyed by outcome. If lasting on-plate marks are wanted later, they'd
  need to be parented to the live plate pose (per-plate mark group transformed
  with the reaction quaternion) — deferred.
